# Atlas Vector Search

[Atlas Vector Search](https://www.mongodb.com/docs/atlas/atlas-vector-search/vector-search-overview/) enables you to perform semantic searches on vector embeddings stored in MongoDB Atlas.
Vector search allows you to find similar documents based on their meaning rather than exact keyword matches, which is essential for building modern AI applications like semantic search, recommendation systems, and RAG (Retrieval-Augmented Generation) applications.

Mongoose provides full support for creating vector search indexes and running vector search queries through the MongoDB aggregation pipeline.

## Creating a Vector Search Index {#creating-index}

To use vector search, you first need to create a vector search index on your collection.
You can do this by calling `schema.searchIndex()` with `type: 'vectorSearch'` and defining the vector fields you want to index.

```javascript
const movieSchema = new mongoose.Schema({
  title: String,
  plot: String,
  // Vector embeddings for the plot, typically generated by an embedding model
  plot_embedding: {
    type: [Number],
    required: true
  }
});

// Define a vector search index
movieSchema.searchIndex({
  name: 'vector_index',
  type: 'vectorSearch',
  definition: {
    fields: [{
      type: 'vector',
      path: 'plot_embedding',
      numDimensions: 1536,  // Must match your embedding model's dimensions
      similarity: 'cosine'  // or 'euclidean' or 'dotProduct'
    }]
  }
});

const Movie = mongoose.model('Movie', movieSchema);
```

### Index Definition Fields

The vector search index definition requires the following fields:

* `type`: Must be `'vector'` for vector fields
* `path`: The field path containing the vector embeddings
* `numDimensions`: Number of dimensions in your vector embeddings. This must match the output dimensions of your embedding model (e.g., 1536 for OpenAI's text-embedding-ada-002, 768 for sentence-transformers)
* `similarity`: The similarity function to use for comparing vectors. See [Similarity Metrics](#similarity) below

### Creating the Index

Once you've defined the vector search index in your schema, Mongoose can automatically create it when your model initializes if you enable the `autoSearchIndex` option:

```javascript
const movieSchema = new mongoose.Schema({
  title: String,
  plot_embedding: [Number]
}, {
  autoSearchIndex: true  // Automatically create search indexes on init
});

movieSchema.searchIndex({
  name: 'vector_index',
  type: 'vectorSearch',
  definition: {
    fields: [{
      type: 'vector',
      path: 'plot_embedding',
      numDimensions: 1536,
      similarity: 'cosine'
    }]
  }
});

const Movie = mongoose.model('Movie', movieSchema);

// The vector index will be created automatically when the model initializes
await Movie.init();
```

Alternatively, you can create the index manually using `Model.createSearchIndexes()`:

```javascript
// Create all search indexes defined in the schema
await Movie.createSearchIndexes();

// Or create a single index
await Movie.createSearchIndex({
  name: 'vector_index',
  type: 'vectorSearch',
  definition: {
    fields: [{
      type: 'vector',
      path: 'plot_embedding',
      numDimensions: 1536,
      similarity: 'cosine'
    }]
  }
});
```

## Querying with $vectorSearch {#querying}

Once your vector search index is created, you can perform vector similarity searches using the `$vectorSearch` aggregation stage.

### Basic Vector Search Query

```javascript
// Generate an embedding for your search query (using your embedding model)
const queryEmbedding = await generateEmbedding('romantic comedy about friendship');

// Perform vector search
const results = await Movie.aggregate([
  {
    $vectorSearch: {
      index: 'vector_index',     // Name of your vector search index
      path: 'plot_embedding',     // Field containing the vectors
      queryVector: queryEmbedding, // Your query embedding as an array of numbers
      numCandidates: 100,         // Number of candidates to consider (should be >= limit)
      limit: 10                   // Number of results to return
    }
  },
  {
    $project: {
      title: 1,
      plot: 1,
      score: { $meta: 'vectorSearchScore' }  // Include similarity score
    }
  }
]);

console.log(results);
// [
//   { _id: ..., title: 'When Harry Met Sally', plot: '...', score: 0.89 },
//   { _id: ..., title: 'The Big Sick', plot: '...', score: 0.85 },
//   ...
// ]
```

### $vectorSearch Options

The `$vectorSearch` stage accepts the following options:

* `index` (required): Name of the vector search index to use
* `path` (required): Field path containing the vector embeddings
* `queryVector` (required): Array of numbers representing your query embedding
* `limit` (required): Maximum number of documents to return
* `numCandidates` (conditional): Required for approximate search. Number of candidate documents to examine. Must be >= `limit`. We recommend setting this to at least 20x your `limit` for better accuracy
* `filter` (optional): MQL filter expression to pre-filter documents before vector search. See [Pre-filtering](#pre-filtering)
* `exact` (optional): Boolean. If `true`, performs exact nearest neighbor search. If `false` or omitted, performs approximate search. See [Approximate vs Exact Search](#ann-vs-enn)

## Similarity Metrics {#similarity}

MongoDB Atlas Vector Search supports three similarity functions for comparing vectors:

### cosine

Measures similarity based on the angle between vectors, independent of magnitude.
Best for: Text embeddings where magnitude doesn't matter.

```javascript
movieSchema.searchIndex({
  name: 'vector_index',
  type: 'vectorSearch',
  definition: {
    fields: [{
      type: 'vector',
      path: 'plot_embedding',
      numDimensions: 1536,
      similarity: 'cosine'
    }]
  }
});
```

**Note**: You cannot use zero magnitude vectors with `cosine`. For cosine similarity with normalized vectors, consider using `dotProduct` instead.

### euclidean

Measures the straight-line distance between vector endpoints.
Best for: When the magnitude and position in space matter, such as spatial data or when vectors aren't normalized.

```javascript
similarity: 'euclidean'
```

### dotProduct

Measures similarity like cosine but considers vector magnitude.
Best for: Normalized embeddings from models like OpenAI's ada-002 where vectors are already unit length.

```javascript
similarity: 'dotProduct'
```

**Note**: When using `dotProduct`, you must normalize vectors to unit length at both index-time and query-time for accurate results.

## Pre-filtering Results {#pre-filtering}

You can use the `filter` option to pre-filter documents before performing vector search.
This is useful when you want to restrict your search to a subset of documents based on metadata.

```javascript
const movieSchema = new mongoose.Schema({
  title: String,
  plot: String,
  plot_embedding: [Number],
  year: Number,
  genre: String,
  rating: String
});

// Vector search with pre-filtering
const results = await Movie.aggregate([
  {
    $vectorSearch: {
      index: 'vector_index',
      path: 'plot_embedding',
      queryVector: queryEmbedding,
      numCandidates: 100,
      limit: 10,
      filter: {
        year: { $gte: 2020 },     // Only movies from 2020 or later
        genre: 'comedy',           // Only comedy movies
        rating: { $in: ['PG', 'PG-13'] }  // Family-friendly ratings
      }
    }
  }
]);
```

The `filter` field accepts standard MongoDB query operators including:

* Comparison: `$eq`, `$gt`, `$gte`, `$lt`, `$lte`, `$ne`
* Logical: `$and`, `$or`, `$not`
* Array: `$in`, `$nin`

**Important**: The fields used in your filter must be indexed in your vector search index definition for optimal performance.

## Approximate vs Exact Search {#ann-vs-enn}

MongoDB Atlas Vector Search supports two types of nearest neighbor search:

### Approximate Nearest Neighbor (ANN)

Default mode. Uses algorithms like HNSW to quickly find approximate nearest neighbors.

* **Pros**: Much faster, scales to large datasets
* **Cons**: May not return the exact nearest neighbors
* **Use when**: You need fast results and approximate matches are acceptable

```javascript
// Approximate search (default)
await Movie.aggregate([
  {
    $vectorSearch: {
      index: 'vector_index',
      path: 'plot_embedding',
      queryVector: queryEmbedding,
      numCandidates: 100,  // Required for approximate search
      limit: 10
      // exact: false (default)
    }
  }
]);
```

### Exact Nearest Neighbor (ENN)

Performs exhaustive search to find the true nearest neighbors.

* **Pros**: Returns the mathematically correct nearest neighbors
* **Cons**: Slower, doesn't scale well to large datasets
* **Use when**: Accuracy is critical and dataset is small to medium sized

```javascript
// Exact search
await Movie.aggregate([
  {
    $vectorSearch: {
      index: 'vector_index',
      path: 'plot_embedding',
      queryVector: queryEmbedding,
      limit: 10,
      exact: true  // Perform exact search
      // numCandidates not required for exact search
    }
  }
]);
```

**Note**: Exact search is available on MongoDB Atlas clusters running v6.0.16+, v7.0.10+, or v7.3.2+.

## Best Practices {#best-practices}

### Choosing numCandidates

The `numCandidates` parameter controls the trade-off between search accuracy and latency:

* **Minimum**: Must be >= `limit`
* **Recommended**: Set to at least 20x your `limit` value
* **Maximum**: 10,000

```javascript
// Good: numCandidates is 20x limit for better accuracy
{
  $vectorSearch: {
    numCandidates: 200,
    limit: 10
  }
}

// Better: For critical searches, use even higher ratios
{
  $vectorSearch: {
    numCandidates: 500,
    limit: 10
  }
}
```

Higher `numCandidates` values improve result quality but increase query latency. Tune this parameter based on your specific accuracy requirements and performance constraints.

### Storing Vector Embeddings

When storing embeddings in MongoDB:

1. **Store as Number arrays**: Use `[Number]` type in your schema
2. **Validate dimensions**: Ensure all embeddings have the same number of dimensions
3. **Don't store embeddings in nested objects**: Vector search requires embeddings at a direct field path

```javascript
// Good: Direct field path
const schema = new mongoose.Schema({
  text: String,
  embedding: [Number]  // Direct field
});

// Bad: Nested object
const schema = new mongoose.Schema({
  text: String,
  vectors: {
    embedding: [Number]  // Vector search won't work on nested paths
  }
});
```

### Combining Vector Search with Other Aggregation Stages

You can combine `$vectorSearch` with other aggregation stages for powerful queries:

```javascript
const results = await Movie.aggregate([
  // 1. Vector search
  {
    $vectorSearch: {
      index: 'vector_index',
      path: 'plot_embedding',
      queryVector: queryEmbedding,
      numCandidates: 100,
      limit: 50  // Get more candidates for filtering
    }
  },
  // 2. Add similarity score
  {
    $addFields: {
      score: { $meta: 'vectorSearchScore' }
    }
  },
  // 3. Filter by minimum similarity threshold
  {
    $match: {
      score: { $gte: 0.7 }  // Only keep results with 70%+ similarity
    }
  },
  // 4. Sort by release date (within similar results)
  {
    $sort: { year: -1 }
  },
  // 5. Limit final results
  {
    $limit: 10
  },
  // 6. Project desired fields
  {
    $project: {
      title: 1,
      year: 1,
      score: 1
    }
  }
]);
```

### Index Quantization for Large Datasets

For very large collections, consider enabling quantization to reduce memory usage:

```javascript
movieSchema.searchIndex({
  name: 'vector_index',
  type: 'vectorSearch',
  definition: {
    fields: [{
      type: 'vector',
      path: 'plot_embedding',
      numDimensions: 1536,
      similarity: 'cosine',
      quantization: 'scalar'  // Reduces memory usage
    }]
  }
});
```

Quantization trades a small amount of accuracy for significant memory savings, allowing you to index larger datasets.

### Monitoring Index Status

After creating a vector search index, it takes time to build.
You can check the index status using `Model.listSearchIndexes()`:

```javascript
const indexes = await Movie.listSearchIndexes();

for (const index of indexes) {
  console.log(`${index.name}: ${index.queryable ? 'ready' : 'building'}`);
}

// Wait for index to be ready
let isQueryable = false;
while (!isQueryable) {
  const indexes = await Movie.listSearchIndexes();
  isQueryable = indexes.find(idx => idx.name === 'vector_index')?.queryable;
  if (!isQueryable) {
    await new Promise(resolve => setTimeout(resolve, 1000));
  }
}
console.log('Vector search index is ready!');
```
